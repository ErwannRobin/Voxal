## What we're looking for

**We welcome:**
- Bug fixes, including platform-specific (macOS, iOS, Android, Web) issues
- Performance and reliability improvements to the WebRTC audio pipeline
- Accessibility improvements
- Documentation fixes and clarifications
- Improvements to the Tauri desktop experience or the Capacitor mobile builds

**Please open an issue first before working on:**
- New features or significant UX changes — we want to discuss the approach before effort is invested
- Changes to the WebRTC topology or host-migration logic — these are complex and need careful coordination
- New dependencies — Voxal intentionally has no bundler and minimal JS dependencies; additions need justification

**Out of scope:**
- Server-side infrastructure (Voxal is serverless by design)
- Changes that require a build step for the `src/` frontend — the web layer must remain a plain static folder

## Getting started

### Prerequisites

- **macOS** is the primary development platform (for Tauri desktop + iOS builds)
- **Node.js** (v18+) and **npm**
- **Rust** (stable toolchain) and **cargo**
- **Xcode** with Command Line Tools (for iOS)
- **Android Studio** with a JDK (for Android)

### First-time setup

```sh
git clone https://github.com/ErwannRobin/Voxal.git
cd Voxal
make install   # installs npm deps and fetches Cargo crates
```

### Development workflow

```sh
make dev          # Tauri desktop app with hot reload (primary dev workflow)
make run-web      # Serve src/ on http://localhost:8080 (web-only testing)
make check        # Fast Rust type-check (run before every commit)
make build-debug  # macOS debug bundle (required once to register voxal:// URL scheme)
```

Run tests before opening a PR:

```sh
npm test              # Playwright web smoke tests + cargo tests
cd src-tauri && cargo test
make check            # fast Rust type-check
```

### Architecture overview

| Layer | Location | Notes |
|---|---|---|
| Frontend | `src/` | Plain HTML/CSS/JS — no bundler, no framework |
| App logic | `src/main.js` | ~5000 lines; all WebRTC, PTT, and room logic lives here |
| Settings (desktop) | `src/settings.html` | Standalone page — duplicates constants from `main.js` |
| Tauri backend | `src-tauri/src/lib.rs` | Rust; IPC commands, global shortcut, deep link |
| iOS wrapper | `ios/` | Capacitor; synced from `src/` via `make cap-sync` |
| Android wrapper | `android/` | Capacitor; synced from `src/` via `make cap-sync` |

> **Important:** `ios/App/App/public/` and `android/app/src/main/assets/public/` are gitignored and regenerated by `make cap-sync`. Always edit files in `src/`, never in those folders.

## Making changes

1. **Fork** the repository and create a branch from `main`:
   ```sh
   git checkout -b your-username/short-description
   ```

2. **Make your changes** in `src/` (or `src-tauri/` for Rust changes).

3. **Sync to mobile** if you changed any `src/` files:
   ```sh
   make cap-sync
   ```

4. **Run `make check`** to verify Rust compiles cleanly.

5. **Test manually** on at least one platform (web via `make run-web` is fine for most changes).

### Platform-specific notes

- **New Tauri IPC commands**: define in `lib.rs` with `#[tauri::command]`, add to `generate_handler![]`, add permissions to `src-tauri/capabilities/default.json`, call via `window.__TAURI__.core.invoke(...)` guarded by `if (window.__TAURI__)`.
- **Platform detection**: always use `if (window.__TAURI__)` / `window.Capacitor?.isNativePlatform()` — never assume a single runtime.
- **`settings.html`** is a standalone page that duplicates constants from `main.js`. If you add a constant to `main.js` that `settings.html` also needs, add it there too.

## Commit guidelines

- Use the **imperative mood** in the subject line: `fix: remove rejoin bar after failed reconnect` not `fixed rejoin bar`.
- Follow the **Conventional Commits** format: `type: short description` where type is one of `feat`, `fix`, `docs`, `refactor`, `chore`.
- Keep commits focused — one logical change per commit.
- Run `make check` before committing if you changed any Rust code.

## Opening a pull request

1. Ensure your branch is up to date with `main`.
2. Fill in the pull request description: what changed, why, and how to test it.
3. Link to any related issue (e.g. `Closes #42`).
4. A maintainer will review your PR. Please allow up to a few days for a response.

PRs that pass `make check` and include a clear description are reviewed faster.

## Reporting bugs

Search [existing issues](https://github.com/ErwannRobin/Voxal/issues) before opening a new one. When filing a bug, please include:

- **Platform** (macOS desktop, iOS, Android, Web browser + version)
- **Steps to reproduce**
- **Expected vs actual behaviour**
- **Voxal version** (visible in About Voxal on desktop, or the Settings modal on web/mobile)

## Reporting security vulnerabilities

Please **do not** open a public issue for security vulnerabilities. See [SECURITY.md](./SECURITY.md) for the private reporting process.
